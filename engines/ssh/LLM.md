## Hives
Set of (hex-)adjacent alveolii. All alveolus should be reachable from any other alveolus in the same hive, moving one tile at a time. No two hives should be adjacent.
## Locales
Translations are merged in memory. `assets/locales` serves as a base, and `src/locales` can override/extend it.
- `assets/locales/*.json`: Base translations.
- `src/locales/*.json`: Source code translations.

## Residential (v1 slice)
- Zoned `UnBuiltLand` can receive `residential:basic_dwelling` as a `project` string; it maps to a `ConstructionTarget` of kind `dwelling` (`construction-state.ts`).
- `BuildDwelling` is a tile-backed construction shell (not an `Alveolus`, not hive-attached) and reuses the `buildAlveolusMarker` hook so engineers can `construct` it like `BuildAlveolus`.
- Completed housing is `BasicDwelling` (`board/content/basic-dwelling.ts`) with a single home slot; characters prefer reserving it over raw residential tiles (`npcs/context/find.ts`).
- Save/load persists dwellings via `SaveState.dwellings` (`DwellingPatch`) applied after `projects` in `game.generate` / `generateAsync`.
- Automatic project creation is driven by `trySpawnResidentialProject` (`residential/demand.ts`) on a ticked `ResidentialDemandTicker` registered on the `Game`.
- Engineers pick **ready construction shells first**, then **foundation** (`hive/engineer.ts` two-pass `findNearest`) so a nearer queued foundation cannot starve a farther `BuildDwelling` that is already material-ready.
- **Shared construction shells**: `ssh/build-site.ts` now defines the structural `BuildSite` contract, **`isStandaloneBuildSiteShell`** (hive-less shells such as `BuildDwelling` vs hive `BuildAlveolus`), installs shared prototype accessors (`installBuildSitePrototype`) on both `BuildAlveolus` and `BuildDwelling`, centralizes material math (`materialRemainingNeeds`, …), and owns phase sync (`registerConstructionMaterialPhaseEffect`). Inspector snapshots and `constructionStep()` operate on that shared shell model.
- **Freight routes (engine)**: `FreightLineDefinition` is an ordered `stops[]` of **`FreightStop`** route steps. Each step has optional `loadSelection` / `unloadSelection` (`GoodSelectionPolicy`) and **either** `anchor: FreightBayAnchor` (bay tile) **or** `zone: FreightZoneDefinition` (radius area) — not both. Gather vs distribute is inferred from geometry (`findGatherRouteSegments` / `findDistributeRouteSegments`: gather = radius zone then bay anchor at the **same** coordinates; distribute = bay anchor pickup that is not a gather unload, then anchor/zone unload). `normalizeFreightLineDefinition` trims ids/coords/policies; `GamePatches.freightLines` / `replaceFreightLine` use **`FreightLineDefinition`**. Longer overview: `docs/freight-lines.md`; handoff / UI gaps: `sandbox/freight-handoff.md`.
- **Segment-specific runtime authority**: Use `gatherSegmentAllowsGoodTypeForSegment` / `distributeSegmentAllowsGoodTypeForSegment` with the active segment in loops. Line-wide `gatherSegmentAllowsGoodType` / `distributeSegmentAllowsGoodType` OR across segments (broad checks). `distributeSegmentWithinRadius` / `distributeSegmentBayTile` take a concrete segment. `freightLineAllowsGoodType` is UI-oriented. Standalone construction delivery/requisition use the `ForSegment` helpers.
- **Construction site scan (distribute radius)**: **`freight/construction-demand.ts`** exposes **`visitStandaloneConstructionSitesForDistributeSegmentAxial`** for callers that need standalone shells in segment radius (e.g. bay requisition). Site predicate: **`isStandaloneBuildSiteShell`** (not hive `BuildAlveolus`). There is no separate NPC `freightDeliver` job; construction supply uses line-vehicle **`zoneBrowse`** (provide branch) / script transfer steps, not a distinct planner job name.
- **Bay requisition**: `augmentFreightBayGoodsRelationsForConstruction` uses **`visitStandaloneConstructionSitesForDistributeSegmentAxial`** and is wired from `StorageAlveolus.workingGoodsRelations` (guarded when `tile.game` is missing — some unit mocks omit `game`).
- **Multi-line freight bays**: `findGatherFreightLines` / `findDistributeFreightLines` match stops touching **gather** vs **distribute route segments**. `StorageAlveolus` (`road-fret`) uses `gatherSegmentAllowsGoodType` for `2-use` provide ads, uses `distributeLinesAllowGoodType` (which delegates to `distributeSegmentAllowsGoodType`) at the bottom of `workingGoodsRelations`, and gates `canTake` so gather-only bays stay closed to convey intake while gather+distribute bays accept goods any distribute segment allows. **On-foot `gather` jobs are not emitted from road-fret bays** — line collection uses wheelbarrow **`vehicleHop`** / **`zoneBrowse`** (loose pickup is `zoneBrowse` with `zoneBrowseAction: 'load'`; walking to the vehicle is a `vehicleHop` prelude via `approachPath`). **Bay anchor** transfers use convey + `VehicleFreightDock` in `freight/vehicle-freight-dock.ts`; scripts live in `assets/scripts/vehicle.npcs` and are invoked through `work.goWork` → `vehicle.*`. The old gather-hut stack (`GatherAlveolus`, `action.type: 'gather'`, `job: 'gather'`) is removed; **save patches** may still list `alveolus: 'gather'`, and `game.ts` maps that legacy key to `freight_bay` when loading.
- **Bay inspector freight UI**: `apps/browser/.../AlveolusProperties.tsx` shows a bay-specific "Lines at this bay" block for `freight_bay` + `road-fret`. Add gather / add distribute calls `createExplicitFreightLineDraftForFreightBay` + `Game.replaceFreightLine`. **Delete** an explicit line from the **line** inspector (`FreightLineProperties`, `Game.removeFreightLineById`; implicit `:implicit-gather:` ids are refused in engine). Locale keys live under `bay.*` / `line.deleteLine.*` in `engines/ssh/assets/locales/*.json`.
- **Construction demand module**: `freight/construction-demand.ts` centralizes the standalone-site axial visitor; bay requisition uses it (no duplicated tile loops).
- **Freight line utility (pure, not wired to begin-service yet)**: `freight/freight-stop-utility.ts` measures zone loose-goods source, zone standalone construction need sink, hive summed stored source, and hive need sink (priority × aggregate `hasRoom`). `freight/freight-line-segment-utility.ts` combines source/sink snapshots with `transferableGoodsMin` and `computeFreightLineSegmentUtility` (distance / travelTime / staleness weights from `engine-rules`: `freightLineUtilityWeights`, `freightLineHiveNeedPriorityWeight`). Vehicle cargo caps are optional TODO hooks only.
- **Downstream freight lookahead**: `freight/freight-stop-utility.ts` also computes `furtherNeededGoods` / `furtherProvidedGoods` / `furtherTransferredGoods` over the remaining stop suffix. The traversal is chronological: a later source can satisfy only a still-later sink, never an earlier one. Current vehicle cargo is projected separately with `projectLoadedGoodsAgainstFurtherNeeds(...)`, which yields retained cargo, extra load need, and safe surplus.
- Debug: `traces.residential?.log(...)` is wired in residential lifecycle hotspots; keep trace sinks as `namedTrace(...)` / `setTraceLevel(...)` rather than assigning bare `console` (Vitest worker serialization can overflow if `console` is assigned into `traces.*`).
## Memoization
All previously commented-out `@memoize` decorators have been rehabilitated and identified with the `// REHABILITATED MEMOIZE` marker.
- **Runtime Guard**: `mutts.reactiveOptions.onMemoizationDiscrepancy` is configured in `debug.ts` to trigger a `debugger` and throw an error if a discrepancy is detected.
- **Test Enforcement**: This detection is enforced in all unit/integration tests (via `test-setup.ts`) and is captured in browser E2E tests via the `console-trap` (initialized in `App.tsx`).
- **`actionDescription` is not memoized**: `npcs/object.ts` derives it from `runningScripts`, and `runningScripts` is intentionally `@unreactive` script-VM state. Treat it as a fresh diagnostic snapshot, not a reactive computed.

## Pathfinding and Reactivity
- Pathfinding must never register reactive dependencies on traversed tiles, borders, loose goods, or scoring targets.
- Any board scan or path search is a transient query, not a reactive derivation.
- The shared pathfinding utilities in `src/lib/utils/pathfinding.ts` are wrapped in `untracked(...)` so callers cannot accidentally subscribe to every visited node.
- Do not put pathfinding-based availability checks behind `@memoize` unless they depend only on stable, coarse invalidation signals.
- **Utility planning** (`population/findNextActivity.ts`): `ActivityPlanningCharacter` includes needs + paths + `resolveBestJobMatch` only — no `carry` / transport field (need scores do not depend on inventory state).
## Advertising / Movement
- Movement selection is deferred in `src/lib/hive/hive.ts` to avoid reactive cycles, so anything that treats a selected partner as an already-created movement is suspect.
- Optional stalled-exchange recovery lives in `src/lib/globals.ts` as `options`; when enabled, hives rescan for stable provide+demand pairs that still have no active `movingGoods` and re-advertise them.
- Convey visuals in `src/lib/npcs/context/work.ts` must use the pre-`hop()` origin snapshot for interpolation; `MovingGood.hop()` mutates `mg.from` to the destination immediately.
- `MovingGood.claimed` prevents two workers from picking up the same movement simultaneously. `conveyStep()` sets it to `true` immediately; `finished()` clears it after re-reserving the next hop; `canceled()` and `cleanupFailedConveyMovement()` also clear it. `aGoodMovement` skips claimed movements.
- Convey paths are **border-to-border across relay tiles**, ending on the demander tile. For non-terminal hops, `conveyStep` **allocates the next gate** before `fulfillMovementSource` (preflight), so a full gate blocks pickup instead of mutating the world first. Multi-movement **cycles** skip preflight for the first movement (the "cycle leader"), freeing one gate slot so subsequent members can allocate. The deferred allocation is retried in the `.finished()` callback.
- In `TrackedMovement` methods, do not mix the closed-over raw object with the runtime receiver for bookkeeping. Keep removal/retracking id-based in hive helpers and use `this` consistently inside `hop()` / `place()` / `finish()` / `abort()`, otherwise reactive proxy receivers can produce stale `movingGoods` bucket skew (`tracked-at-wrong-position`).
- **`conveyStep` / hive rebind**: `Hive.getCanonicalMovement` + `rebindConveyMovementRows` (`npcs/context/convey.ts`) reattach to the canonical `TrackedMovement` (same `MovementRef`, new object after rebind) before/during the convey `MultiMoveStep`. Border allocation recovery (`recoverBorderMovement`) must not clear `claimed` while a worker still holds the convey claim.
- **`recoverTileMovement` logging**: success uses `traces.advertising?.warn(...)` when the channel is assigned; otherwise `console.debug(...)` so `test-setup.ts` (which hooks `console.warn`/`error` and flags `[WATCHDOG]`) does not fail after a successful bookkeeping repair when traces are disconnected.

## Jobs vs work plans (ssh)

In this engine, a **`Job`** is a **planner-facing activity class**: a small discriminated union (`JobType` in `src/lib/types/base.ts`) that `Character.resolveBestJobMatch` / alveolus `nextJob` can return and that `work.goWork` dispatches by `jobPlan.job`. It is **not** every scripted subroutine: vehicle **approach**, **beginService**, and **tile load/provide** are **script phases** (often folded into `vehicleHop` / `zoneBrowse` as optional fields), and only the semantic interactions stay in `Job` / `WorkPlan` as far as the planner is concerned. Internal transfer payloads (`loadOntoVehicle`, …) may still appear as `WorkPlan` shapes for NPC steps/tests but are **not** `Job` values.

## Vehicles
- `VehicleEntity` (in `population/vehicle/entity.ts`) is a first-class game object with its own `uid`, `position`, `storage`, and live `servedLines` references.
- Runtime service state is an optional `service?: VehicleService` holding **references** (`line: FreightLineDefinition`, `stop: FreightStop`, `operator?: Character`) plus a `docked: boolean`. `undefined` ⇒ vehicle is idle.
- Serialized state (`VehicleSerializedState`) stores `servedLineIds` plus `lineId`/`stopId`/`operatorUid`; `VehicleEntity.deserialize` resolves them back to references.
- `Character` holds `operates?: VehicleEntity` and a private `_footPosition?`. `driving` is derived: when onboard there is no private foot position, so `character.position` delegates to `operates.position`; `offboard()` recreates an independent position copy and calls `vehicle.endService()` when any `service` was attached (vehicle stock is unchanged). **`disembarkVehicleKeepingService()`** steps off while keeping `VehicleEntity.service` (bay anchor after dock: line service + `docked`, operator cleared). Freight seams: **`offboardOperatorAfterFreightWorkComplete`** / **`disembarkOperatorLeavingDockedVehicleInService`** (`freight/vehicle-run.ts`). **`Character.onboard()`** checks same-hex using **`axial.round(toAxialCoord(...))`** (not raw float axial keys), consistent with **`VehicleEntity.tile`** and walk-lerp midpoints. Scripted **`abandonAnd`** and a fully drained **`cancelPlan`** call **`releaseVehicleFreightWorkOnPlanInterrupt`** (`freight/vehicle-run.ts`): clear only `service.operator` / `Character.operates`; the vehicle `service` remains attached until an explicit vehicle script completes that service.
- **Vehicle work-plan lifecycle:** `plan.begin` claims vehicle usage (`service.operator` + `character.operates`). Vehicle scripts explicitly end service only at service completion (e.g. `completeVehicleMaintenanceService` for `vehicleOffload`). `WorkPlan.finally` is a safety net that releases the operator/control link while preserving unfinished service, so a character cannot enter self-care / wander / unrelated work while still operating a wheelbarrow.
- **Strict activity correspondence:** non-vehicle work plans and self-care picks must call the vehicle interrupt/release seam before they begin if `character.operates` is still set. A character can only keep `operates` while the running script is a vehicle job for that same vehicle/service.
- **Vehicle release foot sync:** stepping off / disengaging must restore both `_footPosition` and `_tile` / board occupation at the rounded vehicle position (or current foot position if already stepped off). Otherwise the worker can visually remain under the barrow or plan from a stale tile.
- **Unfinished maintenance discoverability:** a `VehicleMaintenanceService` with no `operator` is still an active job offer. `findVehicleOffloadJobApproach` must resume it instead of skipping all serviced vehicles, or the wheelbarrow becomes untakable after an operator release.
- **Maintenance target reachability:** `vehicleOffload` discovery must path-check from the rounded `vehicle.position` to the target tile (`vehicleCanReachMaintenanceTarget` in `freight/vehicle-work.ts`) before advertising load/unload/park, including unfinished maintenance resumes. Axial range alone is not enough; script-time `find.path` is only the execution guard.
- **`walk.until`** (`assets/scripts/walk.npcs`): when the path has no effective movement (empty or already-satisfied/current-tile-only), it now yields a tiny idle **`walk.pause`** step instead of returning immediately. This prevents tight `.npcs` loops from busy-spinning on `walk.until -> return undefined` and hitting `nextStep` throttling. Per effective path step it still moves to the edge midpoint, then `stepOn`s the target tile.
- **Loose-good pickup** (`assets/scripts/vehicle.npcs`): approach the pickup tile with **`walk.untilTileBorder`** (+ **`walk.enter`**), resolve stand position with **`find.pickupPlanLooseGoodPosition`** (offload, after `ensureVehicleOffloadPickupPlan`) or **`find.availableLooseGoodGrabPosition`** (zone **`loadOntoVehicle`**, before `loadOntoVehicleStep`), **`walk.stepUntilGood`**, then **`inventory.effectuate`** / **`vehicle.loadOntoVehicleStep`** (grab only after the intra-tile move). **`inventory.offloadDropBuffer`** is **TS-only** (`npcs/context/inventory.ts`): `vehicle.npcs` pre-walks to a valid wild tile (`canDropLooseHere`) before draining the buffer. The old **`inventory.npcs`** bundle is **removed** (and **`CharacterContract.inventory`** dropped from `assets/scripts/contracts.ts`): a script `offloadDropBuffer` returned **`ScriptExecution`**, the VM advanced past the call before nested work finished, and **`plan.finally`** could **`offboardOperatorAfterFreightWorkComplete`** before loose goods were spawned (see `hive_offload_scenario` integration tests). The TS `@contract` method returns **`ASingleStep | false`**; **`Character.findAction`** may return that **`ASingleStep`** from **`tryTransportOffloadDrain`**, and **`npcs/object.ts` `nextStep` / `abandonAnd` / ponder offload** must schedule it like **`makeRun`’s `ASingleStep` branch**. **`tryTransportOffloadDrain` runs after the `keepWorking` / `findBestJob()` block** so ranked vehicle work (e.g. **`zoneBrowse`**) is not starved by draining the wheelbarrow buffer first.
- **Loose-good offload placement**: `InventoryFunctions.dropAsLooseGood` now jitters the spawned loose-good `position` within the current tile (`axial.randomPositionInTile` + `toWorldCoord(tile.position)`) while still keying the bucket by `character.tile`. This keeps repeated unloads on the same hex from stacking exactly on the center.
- **`canDropLooseHere()` nuance**: it now follows `Tile.isBurdened`, but keeps one maintenance-specific escape hatch: while actively operating a wheelbarrow, the current tile is still considered droppable if the **only** burden is that operated vehicle itself. Deposits, loose goods, projects, residential zoning, or another vehicle on the tile still make the tile invalid.
- Walking characters have **no** personal goods buffer: when not driving, `Character.carry` is **`undefined`**. When driving a line `wheelbarrow`, `carry` is `operates.storage` (`engines/rules/src/content/vehicles.ts`). Use **`requireActiveTransportStorage()`** where transfer code must have vehicle stock. Offload is pending redesign (`sandbox/roadmap-no-character-inventory.md`).
- Line freight uses engine-rules `wheelbarrow` (see `engines/rules/src/content/vehicles.ts`). Planner jobs **`vehicleOffload`**, **`vehicleHop`**, **`zoneBrowse`** are discovered in `freight/vehicle-work.ts` and ranked in `Character.resolveBestJobMatch` (bay anchor unload/load use **convey**, not operator `unloadFromVehicle` / bay load jobs). **`vehicleHop`** may carry `approachPath` / `needsBeginService` preludes; `vehicleApproachStep` only boards; **`vehicleBeginServiceStep`** attaches `service` via **`ensureVehicleServiceStarted(..., { lineId, stopId })`** where ids come from **`pickInitialVehicleServiceCandidate`**: skips lines with no plausible begin-service work: gather lines need at least one reachable loose good of an allowed type within zone radius with vehicle carrier room (carrier is derived from the **vehicle** storage, not from `character.operates` which may not yet be bound); distribute-only lines need standalone construction demand in segment radius, prefers shortest distance to the primary stop target, tie-breaks equal distance by **vehicle stock matching gather/distribute segment goods policies** (avoids vacuous `distributeSegmentAllowsGoodType` when a line has no distribute segments). **`vehicleHopPrepare`** assumes service exists, runs `maybeAdvanceVehiclePastCompletedZoneStop` and `maybeAdvanceVehicleFromCompletedAnchorStop` (anchor advance waits until dock ads + in-flight convey drain). **`vehicleHopDockStep`** docks on **anchor** stops; advance past the anchor is deferred to `maybeAdvanceVehicleFromCompletedAnchorStop` (replaces immediate `advanceVehicleAfterDock` in the dock step). Zone movement uses `freightStopMovementTarget` + `freight-zone-gather-target.ts`. **`loadOntoVehicleStep`** / **`provideFromVehicleStep`** are script-internal transfer steps; they compare **`character.operates?.uid`** to the resolved world vehicle (**mutts**-safe). **`provideFromVehicleStep`** drops via `planDropStored` → `effectuate` (returns a **`DurationStep`**; completion runs on **`finish()`** / tick). Invariants: `freight/vehicle-invariants.ts` (**`assertVehicleServiceOperator`** matches by **`uid`**); **`detachVehicleServiceIfStorageEmpty`** emits **`traces.vehicle?.log`** when service is dropped; optional `//traces.vehicle = console` in `debug.ts`.
- **Vehicle line work ranking**: `Character.vehicleFreightApproachPathLength` scores every planner-visible vehicle job by `approachPath.length`: distance to **claim the vehicle**, not the internal drive leg to the next freight stop / zone/offload target. `vehicleOffload.path` is still mirrored for script compatibility, but scoring reads `approachPath`. Those drive legs are the work itself; counting them in `urgency / (pathLength + 1)` made a loaded line continuation lose to same-tile harvest/transform and stranded wheelbarrows after pickup. `engine-rules` keeps line freight in the ordinary work urgency band (`~2`, like transform/project harvest); maintenance/offload cleanup remains intentionally higher.
- **Unattended line service projection**: when `findVehicleHopJob` considers an already-serviced but operatorless wheelbarrow, it must use `projectedLineStopForVehicleHop(...)`, not the raw `service.stop`. A loaded WB left at a completed gather zone should advertise continuation to the bay/unload stop; otherwise the first worker to reclaim it can get a stale same-zone hop/replan instead of continuing the line.
- **Docked gather unloads**: a docked `VehicleFreightDock` unloads into its own `road-fret` bay through same-tile hive convey. Gather-only bays still reject normal external convey intake via `StorageAlveolus.canTake`, so the source-specific exception is `StorageAlveolus.canTakeFrom(dock, ...)` plus `Alveolus.aGoodMovement` accepting same-tile movement (`from === to === bay tile`). Without both, the WB can dock with stock and then no worker sees a convey job.
- **Zone browse / hop scripts**: line-freight now has an explicit planner/job seam for **`zoneBrowse`** (`freight/vehicle-work.ts`, `assets/scripts/vehicle.npcs`). Zone work is no longer "stay driving and call a TS transfer": the script does **drive to target tile → `walk.enter` → `stepOffVehicleKeepingControl()` → transfer through the linked vehicle storage → `disengageVehicleKeepingService()`**. The temporary offboard state keeps `character.operates` / `vehicle.service.operator` until the transfer finishes, but `character.driving` is false during the foot step; use **`Character.transportStorage` / `requireTransportStorage()`** for this seam, not `carry` (which is still driving-only). **`vehicleHop` at a bay anchor** is different: `vehicleHopDockStep` already calls **`disembarkOperatorLeavingDockedVehicleInService`**, so the script must **not** run that zone-browse prelude again when `dockEnter` is false — gated by **`vehicleHopAnchorDockDisembarked`** (set in `vehicleHopDockStep`, cleared in `vehicleHopPrepare`).
- **`findVehicleApproachJob`** must call `findPathForCharacter(..., punctual: true)`. With `punctual: false`, pathfinding treats the goal as **distance ≤ 1**, so the route can stop on a **neighbor** of the vehicle hex and `Character.onboard()` then fails. Scripts use **`find.pathToVehicle(uid)`** (same start/end and punctual) in `work.npcs` before `vehicleApproachStep`. Optional **`find.isAtVehicle(uid)`** matches the boarding predicate for debugging.
- **Same-hex idle line start is still a seam**: if a worker starts on the exact same hex as an idle served wheelbarrow, `findVehicleHopJob` can currently surface a malformed fresh-line hop (`vehicleHop` without an active line `service`) instead of going through a proper approach/begin-service prelude. Reproductions for line-vs-offload priority should place the worker at least one hex away until that seam is fixed.
- **`work.npcs` dispatch**: `goWork` resolves **`work[jobPlan.job]`** (script exports merged onto `WorkFunctions` in `loadNpcScripts`), then falls back to **`vehicle[jobPlan.job]`** for `vehicleHop` / `zoneBrowse` / `vehicleOffload`. The npc-script surface only allows **one parameter per function**, so the planner/resolution path is not a second VM argument: **`Character.workExecution`** attaches **`path: safePath`** on the `WorkPlan`. Vehicle and alveolus job scripts each own their prelude (e.g. `vehicleHop` preludes inside `vehicle.npcs`, `reachTargetTileLoop` + `assignAlveolusAndPrepare`).
- **`vehicle.npcs`**: `loadNpcScripts` names the bundle from the file path (`vehicle.npcs` → namespace **`vehicle`**). Call vehicle work from **`context.vehicle.vehicleOffload`** (and other `vehicle.*` steps), not `context.work.vehicleOffload` — `work.npcs` only returns `{ goWork, harvest, … }` from `assets/scripts/work.npcs`.
- `Character.carry` is `operates.storage` when `driving`, else `undefined`; inventory natives that move stock call **`requireActiveTransportStorage()`** or guard on `carry` + `freightTransferTime`. `WalkFunctions.moveTo` multiplies duration by `mobilityWalkMultiplier` when driving (wheelbarrow `walkTime` > 1).
- **`provideFromVehicle`** pulls goods from the operated wheelbarrow’s **`VehicleEntity.storage`**, not `freight_bay` alveolus stock. When matching the bay’s wheelbarrow to `character.operates`, compare **`uid`** (not `===`): the character’s reference can be a mutts proxy while `findVehicleEntityAtTile` returns the registered entity.
- There is no `VehicleRunPhase` enum. Whether a vehicle is idle, in transit, or docked is fully expressed by `service` presence + `docked` flag; loading/unloading is an emergent effect of the storage advertising orders while docked.
- **Empty-only begin-line**: planner begin-service (`findVehicleApproachJob` / `findVehicleBeginServiceLeg`) rejects idle wheelbarrows that still carry stock. Existing line continuation is still allowed on non-empty vehicles; only the "start a fresh line service" branch is empty-only.
- **Dock completion trace / park handoff**: `maybeAdvanceVehicleFromCompletedAnchorStop` logs `traces.vehicle?.log('vehicleJob.dock.complete', { ... })` with `outcome: 'advance' | 'end-service' | 'park-next'`. `park-next` means the anchor was the last stop, cargo is empty, line service is ended immediately, and a later planner tick is expected to surface `vehicleOffload{ maintenanceKind: 'park' }`.
- **Park semantics**: `Tile.isBurdened` still counts idle / non-docked wheelbarrows, so a parked empty vehicle still burdens its resting tile tautologically. "Park" means "move the wheelbarrow off the important tile onto a harmless resting hex", not "make the destination tile permanently `!isBurdened` while the vehicle sits there".
- **Debug: `traces.vehicle`** — enable by `setTraceLevel('vehicle', 'log')` / configured traces and inspect with `traces.vehicle.display(count?)`. Selection is logged once when a line-freight job **wins** (`Character.findBestJob`): event `vehicleJob.selected` with job fields + `pathLen`. Milestones (all `traces.vehicle?.log`): `vehicleJob.approach.onboard`, `vehicleJob.beginService`, `vehicleJob.hop.dock` (anchor/bay only), `vehicleJob.hop.zoneReach` (zone stop arrival, calls `vehicle.undock()`), `vehicleJob.load`, `vehicleJob.unload`, `vehicleJob.provide`, `vehicleJob.provideFromVehicle.skippedNoService`, `vehicleJob.offboard.endService`, `vehicleJob.offboard`, `vehicleJob.offboard.dockedKeepService`. Load/unload/provide fire when the work **step** starts (inventory `effectuate` may still be async).
- **`work.goWork` + preloaded wheelbarrow**: while driving, `Character.carry` is the **wheelbarrow `VehicleEntity.storage`**. `goWork` no longer clears foot stock up-front; integration tests that need a deterministic `provideFromVehicle` should either run a real load pipeline first or call `WorkFunctions.provideFromVehicleStep` directly.

## NPC scripts: procedure call vs expression call
- In `.npcs` scripts, **`foo.bar arg`** (no parentheses) is a **statement / procedure call** — its return value goes through `executeProcedure` and is **yielded** to the host loop, so the executor pauses on `ASingleStep` / `ScriptExecution` returns.
- **`foo.bar(arg)`** (with parentheses) is parsed as an **expression call** (`executeCall`); the return value is treated as an inline value and **discarded** at the statement level. A `DurationStep` returned this way will silently never run, and `plan.finally` will fire as if the call had no work pending.
- Rule of thumb: `inventory.offloadDropBuffer`, `vehicle.endParkingService`, `walk.until path`, etc. (anything that may return a step) **must be written without parentheses** when used as a standalone statement. Only put parens when assigning to a variable / using inside an expression (e.g. `kind = vehicle.maintenanceKind()`, `if not inventory.canDropLooseHere() then ...`).

## NPC script loop / infinite-fail
- **`Action infinite fail: … goWork …`** comes from [`npcs/object.ts`](engines/ssh/src/lib/npcs/object.ts) `nextStep` when a script returns no pending step and `findAction` picks the **same** `ScriptExecution.name` twice. The `console.error` includes a **second argument** (subject `name`/`uid`/tile, last `makeRun` `type` + value kind, shallow `nextAction` summary, `lastPlannerSnapshot` / `lastWorkPlannerSnapshot` when present on `Character`). For **`work.goWork`**, `nextAction` also includes **`jobPlan`** (primitive job fields) and **`pathLen`** from the script VM locals when available, so you can compare planner intent with the bound path without dumping reactive graphs.
- If **`lastMakeRun`** is `{ type: "return", valueKind: "undefined" }` for **`work.goWork`**, the whole `goWork` body finished in **one** `makeRun` (no `ASingleStep`). Common for **line-freight** work when there is **no walk** (`pathLen` 0 and no `dockEnter` entry) and hop/dock steps run synchronously; the work planner can still rank the same job best on the next `findAction`, which triggers this guard and **`PonderingStep`** throttling.
- **`traces.npc`** — use configured traces / `setTraceLevel('npc', 'log')`, then `traces.npc.display(count?)` for a compact dump. On infinite fail, `traces.npc?.log` emits `nextStep.infiniteFail` with the same context plus a short `loopTail` (last few `makeRun` entries with value kinds). Helpers live in [`npcs/npc-diagnostics.ts`](engines/ssh/src/lib/npcs/npc-diagnostics.ts).
- **Browser dump helper**: the browser shell exposes **`window.dumpSshDebugState(options?)`** and **`window.dumpSshDebugStateJson(options?)`** from `App.tsx`. They snapshot characters, vehicles, freight lines, the current selection (`selectedUid` defaults to `selectionState.selectedUid`), and optionally `saveState` (`{ includeSaveState: true }`) into JSON-safe data for bug reports when the UI is wedged. Character entries use **`activeTransportStock`** (`character.carry?.stock ?? {}`: vehicle storage only when driving). JSON output is produced via **`cloneValueForDebugJson`** (`debug-game-state.ts`): it unwraps Mutts proxies, avoids enumerating `Game` host graphs (stubbed), stringifies functions, and marks cycles as `"[Circular]"` so DevTools `JSON.stringify` never walks reactive `Eventful.emit` proxies.

## Vitest / integration harness
- **`vehicleOffload` + `drop.stone`**: In some narrow `loadScenario` harnesses, `debugActiveAllocations()` can keep `drop.stone` open for long stretches of virtual time (wheelbarrow still holding stone) while `tests/test-engine/engine.ts` advances the clock. Do not assume “allocations cleared” or “wheelbarrow empty” matches a single `begin(goWork)` unless you use a scenario known to complete the drop leg (see `hive_offload_scenario` “Leak” / Residential-style tests). Planner wiring (`findVehicleOffloadJob` / `findBestJob`) can still be asserted cheaply.
- **`traces`**: Channels are stable and level-gated by `traceLevels` (`log` enables log/warn/assert/error; `warn` enables warn/assert/error; `assert` enables assert/error; `error` enables error only). Call sites use `traces.vehicle.log?.(...)` (no `?.` on the channel); disabled methods stay `undefined`, so argument expressions are not evaluated. Defaults keep `vehicle` at `log` and most other channels at `assert`; raise a channel with `setTraceLevel(...)` when investigating. Configured sinks forward captured/rendered rows as collapsed console groups (`<[channel]> event` title, payload inside via the matching `console.*` method); use `namedTrace('channel', { silent: true })` for test/headless capture without console output. `disconnectAllTraces()` clears cached sinks between tests. Do not assign `console` directly; use `namedTrace('channel', { silent: true })`, `setTraceLevel(...)`, or a small test stub. `namedTrace` snapshots trace args at log time through `trace.ts`, stores JSON-safe rows, dedupes repeated runtime refs, prunes old time-stamped rows by `DEFAULT_TRACE_LOG_LIFETIME`, exposes `trace.read(count?)` for compact YAML-like copy/paste output, and `trace.display(count?)` for `console.log(trace.read(count))`.
- **`Character.operates` setter** (see `population/character.ts`) requires the vehicle to already have **`service.operator` set to that character** (e.g. `vehicle.beginOffloadService(worker)` or line service + `setServiceOperator`). Tests that only do `character.operates = vehicle` without that precondition will fail the assert.
- `vitest.config.ts` sets a **default per-test ceiling** (`testTimeout: 120000` ms) plus `hookTimeout` / `teardownTimeout` (`60000` ms) so a stuck test cannot run indefinitely; stricter limits can still be set per `it(..., { timeout: n })`.
- `tests/test-engine/mocks.ts` must register `vi.mock(...)` at module top level (Vitest hoisting). `test-setup.ts` imports `./tests/test-engine/mocks` first so `engine-rules` never loads before overrides.
- Trace projection is allowlist-first: runtime objects should use `toTrace()` or the structural projectors in `trace.ts` (characters, vehicles, tiles, loose goods, convey movements, freight lines/stops, storage). Plain objects and null-prototype records recurse with depth/width limits; unknown class instances become `$unprojected` placeholders and are never enumerated. Keep heavy fields summarized (path length/from/to, stock totals, ids/coords) rather than dumping full graphs.
- **Script execution errors**: `ScriptExecution.run` attaches source-location diagnostics to `ExecutionError`; `withScripted.makeRun` emits one structured `traces.script` / `script.makeRun.error` entry with subject + running scripts. Avoid reintroducing paired `console.error` calls in both layers.
- When tests override `traces.advertising`, **never** `{ ...console, warn: ... }`: shallow-copying `console` can make Vitest 3.2's worker `postMessage` serialization recurse until `RangeError: Maximum call stack size exceeded`. Prefer a tiny stub (`log/info/debug/error` no-ops + `warn` sink) or a silent `namedTrace`.
- Watchdog tests that manually disturb `Hive.movingGoods` / `activeMovements` must keep allocation guard state in sync: abort real movements when discarding them, or deliberately leave a well-formed orphan for `cancelOrphanedExchangeAllocations`. Shared `test-setup.ts` resets `resetDebugActiveAllocations()` before/after each test to avoid stale allocation tokens leaking across Vitest cases.
- The former `it.fails` BUG block in `convey_bookkeeping_resilience.test.ts` was removed (comment points to git history). The border **source rebind** regression is kept (currently skipped) in `tests/integration/convey_bookkeeping_border_handoff.test.ts` because running it alone can still trigger the same RPC overflow unless Vitest is upgraded.
